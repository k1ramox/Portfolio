#### Pet Lovers' Lounge (a social media for pet lovers)
#### Mobile Responsive: Not implemented due to lack of time for study.
#### Description: I created a web application for pet lovers where users can make accounts and connect with the community. It includes a login and registration system, a digital token currency, user profiles, messaging features, a ranking system, and options to add or remove pets, among other things. I have a total of 15 html files, 6 javascript file, 2 py file app.py and helpers.py, and 1 database file named 'ppl.db'. My database file has five tables, excluding the sequence table: users (which contains user credentials like username, passwords, and status such as following count and more), pets (which holds pet information like name and species), followers (which contains the follower's ID and the ID of the user they follow), activities (which records the user ID of the doer, the target user's ID, the action, the time, and more), and messenger_info (which includes the necessary details for my messaging system, with columns like sender_id, receiver_id, text message, time, and more). In my layout.html, I link all the necessary Bootstrap resources in the head section, along with my webpage icon located at static/paw.png and my custom style.css for additional styling, combining Bootstrap with my own CSS. The webpage title is also set here. In the body section, I include a navbar (top section), a main content area (middle section), and a footer (bottom section). If the user is not logged in, they see three buttons in the navbar: 'Home,' 'Log-in,' and 'Register.' If logged in, they see a search field for looking up profiles or accounts, along with the nav items 'Home,' 'My Profile,' 'Messages,' 'Account,' and 'Log-out.' The main section uses Jinja2 syntax as a block, allowing other HTML templates to reuse layout.html and inject their own content. Lastly, the footer contains a <p> element displaying the year (2024) and an <a> element that shows the text 'Redirect to my Facebook profile' on hover, linking to my Facebook profile with my name inside. In my '/' route or index.html or by clicking the navbutton 'Home', I use Jinja2 to leverage the existing structure from layout.html. The webpage title changes conditionally: if the user is logged in, it displays 'Home'; otherwise, it shows 'Welcome to Pet Lovers' Lounge.' In the main block, there are conditionals: if the user is not logged in, they see a transparent PNG image on the left and a welcome message on the right, along with a 'Join us today' button that redirects to the register.html route. If the user is logged in, the left side features a ranking system showcasing the 'Top 3 Love Token Holders' and a description of the Love Token, while the right side displays activities like who followed someone, added a pet, or gave a love token to a pet. In my 'register.html', I implemented validations on both the server and client sides. For the client side, I included JavaScript that prevents wasting time; if any required field is missing input, the form will not proceed and will instead display feedback indicating that all fields are required. When a username is taken, instead of refreshing the page to show feedback, I use the Fetch API in JavaScript to provide immediate feedback to the user if the username is unavailable or if the passwords don't match. On the server side, I have added validations for security purposes and to prevent minor bugs like the username is case sensitive and I used the function string.lower() to fix that. In my 'login.html', the process is straightforward: it checks if the credentials match what's in the database. Instead of only remembering the user ID, I also store the username in a session variable for my 'profile.html'. This makes it easier to distinguish between my own profile and others', as there are many conditional elements determining what to display when visiting my own profile versus that of other users. In my /profile route, or profile.html, accessible by clicking the 'My Profile' navbar button, users can view their own profile on the left side. This includes their profile picture, name, and if they haven't set a full name yet, their username will be displayed. Users will see statistics such as follower count, following count, number of pets, total love tokens, and love tokens earned (which are gained when someone clicks the 'Give Love' button for one of their pets—note that users can't give love to their own pets). There’s also a 'Set-Up Profile' button that redirects to account.html when pressed. On the right side, a scrollable list displays the pets the user has added. Each pet box contains a picture, pet name, species name, the number of love tokens received, and an 'X' button to remove the pet. This 'X' button triggers a form with an action set to /remove_pet using the POST method. The form includes a hidden input that dynamically sets the value to the pet's ID, which is sent to the remove_pet() function in app.py for processing. Validation is in place to ensure that users cannot modify the pet ID to remove someone else's pet; if they attempt this, an 'error.html' template is rendered. Regarding visiting other user profiles, there is a search field for finding users. In a user's profile, if the visitor hasn’t followed them yet, a 'Follow' button is displayed. This button sends data to the server without refreshing the webpage, using JavaScript’s fetch API. When the button is clicked, it changes to 'Unfollow' and increments the follower count by 1 without a page reload. The 'Follow' button includes an event listener that triggers a fetch function with two arguments: the route /follow, which processes the action in app.py, and an options object containing user data. When sending the user's ID to the server, database commands are executed to update the follower count. Conditional statements check whether the user is already followed; if they are, the action removes the row from the followers table and updates the users table accordingly. If not followed, new data is inserted into the followers table, user columns are updated, and a record is added to the activities table for history purposes. The outcome of this action returns either true or false, which is used in fetch().then() to update the button's label and increment the status. Below the user's name, there’s a clickable text 'Message this user.' On hover, the text decoration changes to underline. Clicking it opens a modal displaying a text area, a send button, and a close button. JavaScript manages the modal’s display, toggling it from none to flex. Server-side validations ensure that empty or space-only inputs do not proceed to SQL commands. Feedback messages are provided, such as "Please input a message" if no input is given, or "Sent" if a valid message is submitted. These feedback messages are positioned absolutely and hidden by default, appearing for two seconds when needed using the setTimeout() function. On the server side, when a user visits another profile, their username is saved in a session variable for easier validation. The 'Set-Up Profile' button is hidden if the profile belongs to someone else, while options like 'Message this user,' 'Follow,' and 'Give Love' (within the pet box) remain visible. The 'GIVE LOVE' button also uses fetch to avoid refreshing the page and also you cannot proceed if you don't have a love token. Moving to the 'Messages' navbar button, this redirects users to /message-inbox, displaying information such as the sender's name, profile picture, message content, and timestamp. Users can click on the message container to redirect to the sender's profile. The backend executes SQL commands to select all rows in the messenger_info table where the receiver's ID matches the logged-in user's ID. If no messages are available, a message will display stating, "There’s nothing to show here because no one has messaged you yet." The returned data is sorted by time in descending order. Users can switch between inbox and sent messages by clicking the corresponding icons in the left sidebar. The sent messages icon redirects to /message-sent, which has a similar implementation to the inbox, but it gathers messages sent by the user. Additionally, I use a border width of 1px and a solid border style to create the container or box around the messages. Next is the 'Account' navbar button. When you click it, you'll see different kinds of options to manage your account: update your profile picture, update your full name, add a pet, change your password, and top up love tokens. In the update profile picture section, I implemented a validation that prevents the process if the picture is not in .jpg format and provides feedback to the user that the file type is invalid. In the update full name section, you will be able to set your full name, especially when you have just joined or registered; you need to update your full name. If you press 'update' without inputting anything in the first name and last name fields (except the middle name), the webpage will notify you that the first name and last name cannot be empty. I also put validations on both the client side and server side regarding the length of these input fields, to prevent special characters, and to capitalize the first letter of the first, middle, and last names. Moving into the '/add_pet' section, here I implemented validations on both the client side (using JavaScript) and the server side (using Flask). I prevent the user from adding a pet if the name exceeds 9 characters and also for the species. Additionally, I prevent submissions whenever there is an empty field, and I validate the image, only accepting jpeg images. If everything is valid, as usual, I insert the gathered data into the database. For my image system, like profile pictures or pet pictures, I store them as files and only save the image filenames in the database. For pet images, I concatenate the user's username with a UUID (a library I used to generate a random unique string) plus the extension '.jpg'. I used a different approach for the user's profile picture since they have unique usernames; I used the username as the filename. Next is the '/change-password' section, which is very straightforward. I also implemented validations on the client side to prevent submission without inputting all the required fields, and on the server side to avoid security issues, since we cannot solely trust client-side validation. If the user disconnects JavaScript from the HTML, they will still not be able to proceed. Lastly, the 'Top-up Love Tokens' feature is just for simulation, like a real webpage that allows buying through digital currency using a wallet, and it is also intended for testing purposes. A newly registered user will receive 5 love tokens as a default setting in the database. The '/logout' process simply clears the session variable.
